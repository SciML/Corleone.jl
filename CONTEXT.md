# Corleone.jl

Corleone.jl formulates dynamic/optimal control problems as a chain of Lux-style layers (mirroring `LuxCore.AbstractLuxLayer`), so the resulting problem can be optimized via `Optimization.jl`.

> Lux-alignment rewrite in progress (feature-controls branch): current scope is `ControlParameter` + `ControlSet` + `SingleShootingLayer`. `MultipleShootingLayer`, `CorleoneDynamicOptProblem`, and the `ext/` extensions still assume the old flat-vector/positional-pairs shape and need their own follow-up design pass.

## Language

**Control parameter**:
A single piecewise-constant control discretization: a name, a time grid, initial values, and bounds. Currently a plain `struct ControlParameter` (`src/local_controls.jl`); the ongoing rewrite turns each one into its own Lux layer so that composing several of them yields a parameter NamedTuple keyed by control name (e.g. `ps.controls = (; NAME = vals1, NAME_2 = vals2)`) instead of today's single flattened, unnamed vector. Its `LuxCore.initialparameters` returns the bare control-values vector directly (no field wrapper), so the container's per-name entry is the raw vector itself, not a nested NamedTuple. The layer keeps its own `name::Symbol` field (in addition to being keyed by name in a container's NamedTuple), so it stays self-describing when used standalone outside a container. It also carries its own `index::Int64` field recording its position in the underlying problem's tunable parameter vector (`problem.p`), replacing today's separate, position-aligned `control_indices::Vector{Int64}` on `SingleShootingLayer`. `index` is supplied by the user at `ControlParameter` construction time (not assigned later by `SingleShootingLayer`); `SingleShootingLayer`'s `controls` keyword takes a collection of fully-formed `ControlParameter` layers (each already carrying its `index`) instead of today's `(idx, ControlParameter)` pairs.
_Avoid_: Control, control variable (without "parameter") when discussing this specific struct/layer.

**ControlSet**:
A `LuxCore.AbstractLuxContainerLayer` holding a NamedTuple of control parameter layers (one per named control) — the same idiom `Lux.Chain`/`Parallel` use, gaining default per-name `initialparameters`/`initialstates` recursion over its sublayers for free. It overrides `initialstates` to additionally merge in the container-level segment-boundary state (not derivable from per-sublayer recursion alone) and defines a custom forward call. Its forward pass, given the live `ps.controls`/`ps.p`, computes the segment grid fresh on every call: `((tspan_1, p_1), ..., (tspan_N, p_N))`, where each `p_i` is the underlying problem's parameter object reconstructed via `SciMLStructures` repacking for that segment. Replaces today's `build_index_grid`/`collect_tspans`/`parameter_vector` closure trio computed once in `SingleShootingLayer`'s `initialstates`. `SingleShootingLayer` calls it first, then feeds the resulting segment grid into `_sequential_solve`. Its bounds are likewise named/structured: `get_bounds` on a `ControlSet` returns `(; NAME = (lb1, ub1), NAME_2 = (lb2, ub2))` instead of today's flattened `control_lb`/`control_ub` vectors spanning all controls. `ControlSet`'s `initialstates` computes segment boundaries once (merging each control's `.t` field with the problem's `tspan`, algorithm unchanged from today's `build_index_grid`/`collect_tspans`) and stores them in `st`; the forward pass only does the live `SciMLStructures` repacking per segment from the current `ps`, not boundary recomputation. `SingleShootingLayer` holds a single `controls::ControlSet` field (no more separate `control_indices` field); its hand-written `initialparameters`/`initialstates` merge `LuxCore.initialparameters/initialstates(rng, layer.controls)` under the `:controls` key alongside the manually-computed `u0`/`p`. Every control's `.index` must be a valid position in the problem's tunable parameter vector — there is no escape hatch for a control that only feeds a loss/constraint without driving the RHS (unlike today's `active_controls` filter); such a quantity should be modeled as a plain optimization parameter instead of a `ControlParameter`.
_Avoid_: index grid, parameter vector, control container (these name the old, now-superseded mechanism, or are pre-naming placeholders).

**Segment grid**:
The tuple of `(tspan_i, p_i)` pairs emitted by a `ControlSet`'s forward pass, one pair per piecewise-constant segment, in time order. Consumed directly by `_sequential_solve(problem, alg, u0, grid, sys)`, which no longer takes separate `param`/`ps`/`index_grid`/`tspans` arguments.
_Avoid_: index grid.
